# Intelligent Segment Selection - LLM-First Architecture

## Overview

This document explains the new LLM-powered segment selection system that replaced the manual rule-based approach.

## The Problem

Previously, the `/coaching/{coaching_id}/transcript` endpoint used **hardcoded rules** to select segments:
- Filler word detection: Checked against `{'um', 'uh', 'like', 'so'...}`
- Pace issues: Simple thresholds (`WPM > 180` or `WPM < 100`)
- Low confidence: `avg_confidence < 0.7`
- Improved audio: Just removed filler words with regex

**This was suboptimal because:**
- Missed subtle issues (monotone delivery, wrong emphasis, unnatural pitch patterns)
- Couldn't prioritize segments by learning value
- Didn't leverage the rich coaching analysis already generated by Claude
- Improved audio wasn't actually improved (just filler removal)

## The Solution: LLM-First Thinking

Instead of app → rules → LLM, we now do **LLM → LLM → app**:

### Architecture Flow

```
1. Audio Processing (existing)
   ├─ Transcription with timestamps
   ├─ Prosody analysis (pitch, intensity, pauses)
   └─ Claude coaching feedback (critique + improved SSML + guidance)

2. Intelligent Segment Selection (NEW)
   ├─ Input: Transcript + coaching feedback + prosody summary
   ├─ LLM Task: Select 4-6 most pedagogically valuable segments
   ├─ Output: Structured JSON with segments + improvements
   └─ Claude decides what's important, not hardcoded rules

3. Audio Generation
   ├─ Extract original audio for selected segments
   ├─ Generate improved audio using Claude's SSML
   └─ Return to user with coaching tips
```

## Key Files

### 1. `prompts/segment_selection_prompt.md`
**Purpose:** Prompt template for segment selection

**Key aspects:**
- Asks Claude to select pedagogically valuable segments
- Provides full context (transcript, coaching feedback, prosody)
- Requests structured JSON output with improvements
- Emphasizes learning value over rule-based severity

### 2. `services/intelligent_segment_selector.py`
**Purpose:** Core service for LLM-based segment selection

**Key functions:**
- `format_transcript_with_timing()` - Formats transcript for Claude
- `create_segment_selection_prompt()` - Builds the prompt
- `select_segments_with_claude()` - Calls Claude API
- `enrich_segments_with_metadata()` - Adds computed fields

**Example output:**
```json
{
  "segments": [
    {
      "segment_id": 1,
      "time_range": {"start": 12.5, "end": 18.3},
      "selection_reason": "Demonstrates excessive filler words and monotone delivery",
      "original_text": "So um I think that we should like consider this",
      "improved_text": "I think we should consider this",
      "improved_ssml": "<prosody rate='95%'>I <emphasis>think</emphasis> we should <emphasis level='strong'>consider</emphasis> this.</prosody>",
      "primary_issues": ["filler-words", "monotone", "weak-emphasis"],
      "coaching_tip": "Remove fillers and emphasize action words...",
      "priority": "high"
    }
  ],
  "selection_summary": "Selected 5 segments covering main challenges..."
}
```

### 3. `services/segment_generator.py` (Updated)
**New function:** `generate_segments_with_audio_intelligent()`

**What changed:**
- Uses Claude to select segments (instead of `select_interesting_segments()`)
- Generates improved audio from Claude's SSML (not just filler removal)
- Leverages full coaching context

**Backward compatibility:** Old `generate_segments_with_audio()` still exists as fallback

### 4. `api/main.py` (Updated)
**Endpoint:** `GET /coaching/{coaching_id}/transcript`

**What changed:**
- Checks if coaching feedback exists
- Uses `generate_segments_with_audio_intelligent()` when available
- Falls back to rule-based if coaching feedback missing
- Fully backward compatible

## Why This Is Better

### 1. **Contextual Understanding**
Claude sees the full picture:
- Full transcript with timing
- Previous coaching analysis it generated
- Prosody patterns (pitch, intensity, pauses)
- Can identify complex patterns humans might miss

### 2. **Pedagogical Prioritization**
Claude selects segments based on:
- Learning value (where improvement has most impact)
- Variety (different types of issues)
- Self-contained units (5-15 seconds)
- Mix of issues + good examples

### 3. **Genuine Improvements**
Improved audio is based on:
- Claude's SSML with proper prosody tags
- Correct emphasis patterns (not just filler removal)
- Natural pitch and pace adjustments
- Grounded in actual speech coaching principles

### 4. **Consistency**
The same AI that:
- Analyzed the speech
- Generated coaching feedback
- Now selects practice segments
- Creates coherent, aligned recommendations

## Cost & Performance

**API Calls:**
- 1 Claude call for coaching feedback (already existed)
- 1 additional Claude call for segment selection (Sonnet, ~$0.003-0.01/request)

**Model Used:**
- Coaching feedback: `claude-opus-4-5` (high quality analysis)
- Segment selection: `claude-sonnet-4-5` (cost-efficient, structured output)

**Response Time:**
- Segment selection: ~2-5 seconds
- Cached for subsequent requests with same `max_segments`

## Usage

### As a standalone script:
```bash
python services/intelligent_segment_selector.py \
  path/to/coaching_analysis.json \
  path/to/coaching_feedback.txt \
  6
```

### From code:
```python
from services.intelligent_segment_selector import select_segments_with_claude

result = select_segments_with_claude(
    coaching_analysis=analysis_dict,
    coaching_feedback=feedback_text,
    max_segments=6
)

segments = result['segments']
```

### Via API:
```bash
curl -H "Authorization: Bearer $TOKEN" \
  "https://api.example.com/coaching/coach_123/transcript?max_segments=6"
```

## Testing

To test the intelligent selection without full pipeline:

```bash
# Set API key
export ANTHROPIC_API_KEY="your-key"

# Run on existing analysis
python services/intelligent_segment_selector.py \
  output/analysis/sample_coaching_analysis.json \
  output/coaching/sample_coaching_feedback.md \
  6
```

This will print selected segments and save to `intelligent_segments.json`.

## Migration

**Existing users:** No changes needed. The endpoint is backward compatible.

**New features enabled when:**
- Coaching feedback was generated (not skipped with `--skip-coaching`)
- ANTHROPIC_API_KEY is set
- Otherwise, falls back to rule-based selection

## Summary

This refactor exemplifies **LLM-first thinking**:
- Don't build rules when the LLM already understands the domain
- Leverage context the LLM already has (coaching feedback)
- Let the LLM do what it's good at (understanding, prioritizing, teaching)
- Use structure and prompts to get reliable output

The result: Better segment selection, more useful improvements, and a system that leverages AI throughout the pipeline cohesively.
